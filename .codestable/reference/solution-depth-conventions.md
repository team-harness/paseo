# 方案深度约定

选实现方案时——缩范围、挑实现手段、定实现深度——的反射检查。与 `shared-conventions.md` 第 7 节结构反射检查互补：那节截"往复杂漂"的结构膨胀，这节截"往省事漂"的实现降级。

## 方案深度 pre-pass

AI 默认会往"最省力那条"漂——最小闭环、fake / mock / stub、正则凑逻辑、占位实现——再包装成"合理的第一步"。这一节把漂移截在选型那一刻。

**核心是场景适配，不是求最小也不是求最全。** 同一决策既可能因省事而降级（偷工），也可能因求稳而镀金（过度）。判据只有一个：这个场景该做到什么程度——看价值、正确性要求、风险、生命周期，不看哪条最好写。

| 触发场景                                      | 停下来问自己                                                                          |
| --------------------------------------------- | ------------------------------------------------------------------------------------- |
| 想把范围缩到"最小闭环 / MVP / 第一版只做这些" | 是"最窄的端到端路径"，还是只是"最容易的一条"？这个场景的价值 / 风险容得下这次缩减吗？ |
| 想用 fake / mock / stub 替代真实实现          | 被替掉的是"外部边界"（可以），还是"本场景的核心逻辑"（不行）？替身什么条件下转正？    |
| 想用正则 / 字符串拼接凑本该结构化处理的逻辑   | 一次性 / 边界解析（可接受），还是核心逻辑在赌输入格式？                               |
| 想抄一版"够跑就行"的实现                      | 这段代码的生命周期：一次性验证，还是长期维护的资产？                                  |

## 场景锚点

不是穷举清单，是校准方向的锚点；具体场景对照着判，别当模板套。

| 倾向做实（别降级）             | 倾向可降级（替身 / 最小闭环合理）        |
| ------------------------------ | ---------------------------------------- |
| 核心业务逻辑、正确性关键路径   | 外部边界输入 / 输出（平台 API、webhook） |
| 长期维护的资产、公共契约       | 一次性验证、探索性原型                   |
| 用户直接依赖的产出             | 依赖未就绪、先跑通架构假设               |
| 错了代价高（数据、资金、安全） | 错了代价低、易回滚                       |

## 停下来之后

降级本身不是错，默认降级、不论证场景才是错。若选简化 / 替身方案，在 design / 迭代记录里显式写清：

- 候选：完整版 vs 简化版。
- 本场景选它的理由——**不接受"更简单 / 更快"作唯一理由**，要落到价值 / 风险 / 生命周期。
- 若是替身（fake / mock / 正则 / 占位）：转正条件——什么时候、满足什么换成真实实现。

并入既有的 design 整体确认，不新增停顿。不许因省事默认最小，也不许因求稳默认镀金。默认动作是：按场景论证深度，再落方案。
